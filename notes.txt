You're on the latest version — @playwright/cli 0.1.14 is both what's installed and the newest on npm. Here's what it does.

First, clear up a naming collision
There are two different "Playwright CLIs", and people constantly confuse them:

What it is	Command
playwright (the test runner)	Runs your .spec.ts files, codegen, show-report	npx playwright test
@playwright/cli ← this one	An agent-driven live browser controller	playwright-cli ...
This skill is about the second one. It's a fairly new tool designed so that an AI (or you) can drive a real browser one command at a time, see what happened, and decide the next step. It is not a test runner.

The mental model: snapshot → ref → act
This is the one idea that makes everything else click. You don't write CSS selectors up front. Instead the loop is:

Snapshot the page — you get an accessibility tree where every interactive element has a short ref like e15.
Act on a ref — click e15, fill e3 "...".
The browser returns a fresh snapshot automatically, so you see the result and pick the next ref.

playwright-cli open https://playwright.dev   # launch + navigate
playwright-cli snapshot                       # see refs: e1, e2, e15...
playwright-cli click e15                       # click the search box (a ref from the snapshot)
playwright-cli type "page.click"
playwright-cli press Enter
A snapshot looks like this — note the human-readable refs:


### Page
- Page URL: https://example.com/
- Page Title: Example Domain
### Snapshot
- heading "Example Domain" [ref=e2]
- link "More information..." [ref=e5]
So you'd then just playwright-cli click e5. No guessing selectors.

What it can actually do — grouped with examples
Interact with the page — the everyday verbs:


playwright-cli fill e5 "user@example.com" --submit   # fill + press Enter
playwright-cli click e3
playwright-cli select e9 "option-value"
playwright-cli check e12        # / uncheck
playwright-cli hover e4
playwright-cli drag e2 e8
playwright-cli upload ./document.pdf
Navigate & manage tabs:


playwright-cli goto https://example.com
playwright-cli go-back            # go-forward, reload
playwright-cli tab-new https://example.com/other
playwright-cli tab-select 0
Read the page / pull data out — eval runs JS, --raw strips everything but the result so you can pipe it:


playwright-cli eval "document.title"
playwright-cli eval "el => el.getAttribute('data-testid')" e5   # inspect a specific element
playwright-cli --raw eval "JSON.stringify([...document.querySelectorAll('a')].map(a=>a.href))" > links.json
Inspect & debug like DevTools:


playwright-cli console          # console messages (filter: console warning)
playwright-cli requests          # network log; request 5 for one in detail
playwright-cli tracing-start    # ... tracing-stop → opens in trace viewer
Mock the network — force errors or fake responses without touching the backend:


playwright-cli route "**/*.jpg" --status=404
playwright-cli route "https://api.example.com/**" --body='{"mock": true}'
Manage auth & storage — capture a logged-in session once, reuse it:


playwright-cli state-save auth.json      # save cookies + localStorage
playwright-cli state-load auth.json
playwright-cli cookie-set session_id abc123 --httpOnly --secure
playwright-cli localstorage-set theme dark
Capture artifacts:


playwright-cli screenshot --filename=page.png   # or screenshot e5 for one element
playwright-cli pdf --filename=page.pdf
playwright-cli video-start demo.webm            # ... video-stop
Choose your browser / session — Chromium, Firefox, WebKit, Edge; even attach to a browser you already have open:


playwright-cli open --browser=firefox --persistent
playwright-cli attach --cdp=msedge              # drive an already-running Edge
playwright-cli -s=mysession open example.com     # named parallel sessions
The feature your cursor is on — and why it matters here
The line you highlighted is the real point of this tool:

Every action you perform with playwright-cli generates corresponding Playwright TypeScript code.

When you playwright-cli click e5, the output includes the equivalent line like await page.getByRole('link', { name: 'More information' }).click();. So the CLI doubles as a live, verified code generator: you drive the real app, and the commands hand you back stable, role-based locators you can paste into a spec.

That's exactly why this repo's CLAUDE.md says to "use the Playwright CLI skill to inspect the live app and capture stable selectors before writing locators — never invent a selector you have not verified." The workflow is:

Explore the live app with playwright-cli (snapshot → act).
Copy the verified locators it emits, or generate one explicitly:

playwright-cli generate-locator e5 --raw
Paste into a page object, following the planner → generator → healer loop.
It also powers test debugging — npx playwright test --debug=cli pauses a failing test and prints a session name you can playwright-cli attach <session> to, then poke at the live paused page to find why a locator broke (see playwright-tests.md).

One-line summary
playwright-cli (v0.1.14) is an interactive, snapshot-driven browser remote control that an agent can use to explore, test, mock, debug, and capture web apps — and because every command emits the matching Playwright TypeScript, it's the bridge between "poking at the live app" and "writing a real, durable spec."
